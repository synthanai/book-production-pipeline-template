## ðŸ“Š Document Hierarchy

**Authority levels:**
1. **Masters** - Highest authority, most comprehensive
2. **Architectures & Blueprints** - Structural specifications
3. **Guides** - Detailed how-to documents
4. **References** - Quick lookup tables

**When documents conflict:** Higher tier wins.

## ðŸ”„ Workflow

### Phase Start
1. Review research findings (from Phase 2)
2. Determine document needs for your project
3. Create document outline and hierarchy

### During Phase
1. Write master documents first (foundation)
2. Create architectures and blueprints (structure)
3. Develop detailed guides (execution)
4. Build reference materials (lookup)
5. Establish quality frameworks (validation)
6. Cross-validate all documents

### Phase End
1. Final validation of all prep docs
2. Create master index/table of contents
3. Archive old versions
4. Update [../metrics/trackers/PREP-PHASE-TRACKER.md](../metrics/trackers/PREP-PHASE-TRACKER.md)

---

## ðŸŽ¯ Best Practices

1. **Start with masters** - Foundation before details
2. **Cross-reference** - Connect related documents
3. **Version control** - Track major changes
4. **Validate thoroughly** - Prep doc errors multiply in writing
5. **Keep updating** - Prep docs evolve as you write
6. **Don't over-prep** - Diminishing returns exist

- Non-fiction with multiple systems
